Phase 1: Guardrails First

The Harness Engineer translates team decisions into always-loaded steering context.

product.md

Domain rules, data model, status transitions, triage logic, scope boundaries

tech.md

Stack: Node 22, TypeScript strict, Express, Prisma, React, Vite, Tailwind, Vitest

structure.md

Monorepo layout, naming conventions, component patterns

AGENTS.md

Strict TDD, BDD style, mandatory workflow, engineering principles, YAGNI

Every Kiro session operates within these boundaries. The AI cannot deviate.

Phase 2: The Specification Cascade

Each role produces a spec. Each spec informs the next layer down:

  • Feature Analyst → requirements.md (EARS)
  • Architect → architecture.md
  • API Designer → api-spec + openapi.yaml
  • Quality Engineer → test-cases.md
  • UI/UX Designer → ui-spec.md
  • Harness Engineer → DESIGN.md + steering

Phase 3: Feature Specs

The Harness Engineer scaffolds implementation specs from team artefacts:

  • database-schema — Prisma models
  • dispute-api — REST endpoints
  • triage-engine — rule evaluation
  • ops-dashboard — dispute list UI
  • dispute-detail-view — case detail
  • create-dispute-form — multi-step form
  • lifecycle-guard — status transitions

Each spec: requirements.md → design.md → tasks.md

The Harness Engineer

The bridge between the specification team and the AI agent.

Day 1: Setup

  • Translate team artefacts into steering files
  • Configure specs with #[[file:...]] references
  • Build hooks for automated quality checks
  • Create skills for repeatable patterns
  • Ensure all team output is Kiro-readable

Day 2: Build

  • Feed specs to Kiro, monitor output quality
  • Run tasks (individually or in parallel)
  • Refine steering when results are inconsistent
  • Debug with #terminal context
  • Adjust hooks as patterns emerge

Not writing application code. Wrangling the agent — the new software engineering role.

Skills & Hooks

Skills

skill-creator

Meta-skill: build & iterate on new skills with eval loops

EARS compliance, keyword checks, REQ numbering, vague language detection

revealjs

Generate presentations like this one

Hooks

update-docs-on-stop

Trigger: Stop

At session end, reviews work and auto-updates AGENTS.md, product.md, structure.md, tech.md if materially out of date.

Skills compound across sessions. Hooks close the feedback loop — docs never go stale.

Key Takeaways

  • Structure enables speed — steering docs aren't overhead, they're why Kiro moves fast without breaking things
  • Specs are executable — EARS requirements map directly to test cases and implementation
  • Separation of concerns — specialists produce documents, the Harness Engineer translates them into a harness
  • Skills compound — build once, use across every session
  • Hooks close the loop — automated doc updates keep the baseline current
  • The cascade beats waterfall — iterate within each layer before flowing down
